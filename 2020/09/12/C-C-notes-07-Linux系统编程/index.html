Linux下的目录结构

/root /bin /boot /dev /etc /home /var /lib /usr /media

  • /bin :binary, 二进制文件, 常用的shell命令. 如 ls, rm, cp
  • /sbin :super user系统管理员使用的系统管理程序. 如 ifconfig, halt, shutdown, reboot 等
  • /dev :device. Linux下一切皆文件, 这里是 硬盘、显卡、显示器; 字符设备文件、块设备文件等
  • /lib :linux运行时需加载的一些动态库. 如 libc.so、libpthread.so等
  • /mnt :手动挂载目录. 如U盘
  • /media :外设的自动挂载目录,如光驱
  • /root :超级用户root的家目录
  • /usr :unix system resource – 类似于Windows的program files目录
    • include 目录内存放头文件, 如: stdio.h、stdlib.h、string.h、pthread.h
    • games 目录内有小游戏, 如: sl小火车游戏
  • /etc :存放系统级别的配置文件
    • /etc/passwd
      • man 5 passwd可以查看passwd文件的格式信息
    • /etc/group
      • man 5 group可以查看group文件的格式信息
    • /etc/profile
      • 系统的配置文件. 修改该文件会影响这个系统下面所有的用户
  • /opt :安装第三方应用程序
    • 比如安装了oracle 数据库一般就在这个目录下
  • /home :linux操作系统所有用户的家目录
  • /tmp :存放临时文件

软链接和硬链接 - ln

  • ln :为保持有效性,**源文件应该用绝对路径**来创建软链接,否则会因为移动而无法访问
    • ln -s 源文件(夹)名 目标软链接名称
    • ln 源文件(夹)名 硬链接名称
      • 注意:前面没有参数 -s 了.
      • 硬链接文件可以移动, 但硬链接不能跨分区
      • 硬链接不能建立在目录上.
      • 每创建一个硬链接,则硬链接计数加1, 每删除一个硬链接,则计数位减1.
  • ls -i 文件名 可以查看文件的i节点(inode)
  • stat 文件名 可以查看i节点信息

用户权限、用户和用户组 - chmod

  • 文字设定法 语法: chmod [who][+|-|=][mode] 文件名
    • 操作对象: [who]
      • u :用户(user)
      • g :同组用户(group)
      • o :其他用户(other)
      • a :所有用户(all)这是默认的选择
    • 操作符: [+|-|=]
      • + :添加权限
      • - :取消权限
      • = :赋予给定的权限并取消其他权限
    • 权限:[mode]
      • r :读
      • w :写
      • x :执行
    • 例子:
      • chmod ug+wr file.txt
      • chmod u+x,o+w file.txt
  • 数字设定法 语法: chmod [+|-|=][mode] 文件名
    • 操作符: [+|-|=]
      • + :添加权限
      • - :取消权限
      • = :赋予给定的权限并取消其他权限(默认为=)
    • 数字表示的含义: [mode]
      • 0 :没有权限(-)
      • 1 :执行权限(x)
      • 2 :写权限(w)
      • 4 :读权限(r)
    • 例子:
      • chmod 664 file.txt

修改文件所有者和所属组 - chown chgrp

  • 语法: chown 文件所有者:文件所属组 文件名
  • 语法: chgrp 文件所属组 文件名

寻找查询命令 - find

  • 按文件名查找: 使用参数 -name

    • find 路径 -name “文件名”
      • find /home -name “*.c” 支持模糊查找

  • 按文件类型查找: 使用参数 -type

    • find 路径 -type 类型(见下)
      • f :普通文件类型(不用符号“-”)
      • d :目录
      • l :符号链接
      • b :块设备文件
      • c :字符设备文件
      • s :socket文件
      • p :管道文件
    • find 路径 -type f

  • 按文件的大小查询: 使用参数 -size

    • find 路径 -size 范围
      • 范围
        • 大于: +表示 如: +100k
        • 小于: -表示 如: -100k
        • 等于: 不需要添加符号 如: 100k
      • 大小
        • M 必须大写, 如 (100M)
        • k 必须小写, 如 (20k)
        • c 表示字节数
      • 例子:
        • find ~/ -size +100k
        • find ~/ -size +50k -size -100k

  • 按文件日前查找: 使用参数 -ctime|mtime|atime

    • 创建日期: -ctime -n/+n
      • -n :n天以内
      • +n :n天以外
    • 修改日期: -mtime -n/+n
    • 访问日期: -atime -n/+n

  • 按深度: 使用参数 -maxdepth|mindepth

    • 参数 -maxdepth n(层数) :搜索n层以下的目录,搜索的层数不超过n层
    • 参数 -mindepth n(层数) :搜索n层以上的目录,搜索的层数不少于过n层

  • 高级查找: find path -name “*.c” -exec (shell命令) {} ;

    ​ find path -name “*.c” | xargs (shell命令)

    • 例: 查找指定目录下所有子目录,并列出目录中文件的详细信息
      • find ./ -type d -exec (使用shell命令) {} ;
        • find ./ -type d -exec ls -l {} ;
      • find ./ -type d -ok (使用shell命令) {} ;
        • find ./ -type d -ok ls -l {} ;
        • 注意: {}中间不能有空格, 但是和 \ 之间有空格
        • ok比较安全(因为会提示),特别是在执行rm删除文件的时候
      • find ./ -type d | xargs (使用shell命令) 优先考虑使用此种方法,因为不使用缓冲
        • find ./ -type d | xargs ls -l

grep命令

  • grep -r (有目录) “查找的内容” 搜索的路径

    • -r 参数: 若是目录,则进行递归搜索
    • -n 参数: 显示该查找内容所在的行号
    • -i 参数: 忽略大小写进行查找
    • -v 参数: 显示不含有某字符串

    例: 搜索当前目录下包含 hello world 字符串的文件

    • grep -r -n “hello world” ./ 显示行号
    • grep -r -i -n “hello world” ./ 忽略大小写

find和grep命令结合使用

  • 先使用find命令查找文件,然后使用grep命令查找哪些文件包含某个字符串
    • find . -name “*.c” | xargs grep -n “main”

tar工具

  • 相关参数

    • z :用gzip来压缩和解压缩文件
    • j :用bzip2来压缩和解压缩文件
    • c :create, 创建新的压缩文件,与 x 互斥使用
    • x :从压缩文件中释放文件, 与 c 互斥使用
    • v :详细报告tar处理的文件信息
    • f :指定压缩文件的名字
    • t :查看压缩包中有哪些文件

  • 压缩

    • tar -cvf 压缩包名字.tar 原材料(要打包压缩的文件或者目录)
    • tar -zcvf 压缩包名字.tar.gz 原材料(要打包压缩的文件或者目录)
    • tar -jcvf 压缩包名字.tar.bz2 原材料(要打包压缩的文件或者目录)

  • 解压缩

    • tar -xvf 已有的压缩包名字.tar
    • tar -zxvf 已有的压缩包名字.tar.gz
    • tar -jxvf 已有的压缩包名字.tar.bz2

    解压缩到指定目录: 添加参数 -C (大写的)

    ​ tar -zxvf test.tar.gz -C ./mytest(这是指定的目录)

  • 查看压缩包中有哪些文件

    • tar -tvf test.tar

rar工具

  • 使用前需安装rar工具
    • Sudo apt-get install rar
  • 压缩: 如果是**目录,要使用参数 -r**. 打包生成的新文件不需要指定后缀
    • rar a -r 包名 原材料(要打包压缩的文件或者目录)
  • 解压缩:
    • rar x 包名 要解压的目录(如不指定,则解压缩到当前目录)

zip工具

  • 压缩: zip -r 压缩包名 要压缩的文件(文件和目录)
    • 压缩目录时,才要使用参数 -r
    • 可以不指定压缩包的后缀名
  • 解压缩: unzip xxx.zip -d 解压目录
    • 解压到指定目录才使用参数 -d. 若目录不存在,则会创建

内存3G-4G区间为**内核区**,包含: 内存管理、进程管理、设备驱动管理、VFS虚拟文件系统. 内核空间是受保护的,用户不能对该空间进行读写操作,否则会出现段错误.

阻塞和非阻塞

阻塞和非阻塞是文件的属性还是read函数的属性?

经过验证,read()函数读普通文件是非阻塞的. read()函数在读完文件内容后,若再次read,则read()函数会立刻返回, 表明read()函数读普通文件是非阻塞的.

经过测试,read()函数读终端设备文件(/dev/tty)是阻塞的. 标准输入STDIN_FILENO. 管道(pipe)和套接字(socket)也是默认阻塞的

阻塞和非阻塞不是read()函数的的属性,而是文件的属性.

  • stat() / lstat() :获取文件信息

    ​ 可以通过 man 2 stat 看struct stat {} 结构体组成

    ​ 其中重要的成员域有

    ​ uid_t st_uid; //用户ID

    • 其中文件读写权限的判断

    • 文件类型的判断 S_IFMT 0170000 掩码, 过滤 st_mode中除了文件类型以外的信息

进程相关概念

程序 和 进程 的区别

程序: 是编译好的二进制文件,存放在磁盘上,占用的是磁盘空间,是一个静态的概念. (就像剧本)

进程: 是一个启动的程序,需要占用系统资源,如内存、cpu、终端等.(就像唱戏,要占用舞台、灯光、人等)

同一个程序可以在多个终端执行,类似于同一台戏可以在多个舞台演出.

每启动一个程序,都会有一个进程PID,即使是相同的程序,多次启动,也会有多个不同的PID.

并发 和 并行

CPU会将一个大的时间段分成多个小的时间片,让进程轮流使用cpu的时间片.

并发: 在同一个时间段内,在同一个cpu上,同时运行多个程序.

并行: 两个或者两个以上的程序同时在同一时刻发生(需要多颗CPU或者多核)

  • 描述控制终端的信息
  • 当前工作目录(Current Working Directory)

​ 注意: 子进程的地址空间没有变化,子进程的PID也没有变化.

​ 返回值: 若成功,则不返回,不会再执行execl()函数后面的代码; 若失败,会执行execl()后面的代码,可以用 perror()打印错误原因.

为什么要进行进程资源的回收

​ 如果父进程不回收资源,会导致即使子进程已死亡,也依然会占用部分资源(因为子进程自己可以回收内存用户区资源,但是不能回收内存内核区资源).当创建的子进程足够多时,会导致系统由于资源不足而系统卡住了. 所以,需要父进程去回收子进程的资源.

孤儿进程

​ 若子进程的父进程已经死掉,而子进程还活着,这个进程就成了孤儿进程

​ 为了保证每个进程都有一个父进程,孤儿进程会被 init进程领养. init进程成为了孤儿进程的养父进程,当孤儿进程退出之后,由 init进程完成对孤儿进程的资源回收

僵尸进程

子进程先退出, 父进程没有完成对子进程的资源回收,此时子进程就变成了僵尸进程.

通过命令行: ps ajx 结果显示<defunct>的,就是僵尸进程.

如何解决僵尸进程:

​ 不能使用 kill -9 杀死僵尸进程, 原因是僵尸进程已经是一个死掉的进程,无法接受并处理信号了.

​ 应该用**杀死僵尸进程的父进程**的方法来解决僵尸进程.

​ 原因是杀死父进程,可以导致僵尸进程被 init进程领养, 这样 init进程就会回收僵尸进程的资源.这个动作相当快.

进程回收函数 - wait() 和 waitpid()

  • wait()函数 - 只能在父进程中调用,完成对子进程的回收.这是一个阻塞函数

    • 函数原型

      ​ pid_t wait(int *status); : 如果不想查status的状态,可以传NULL

    • 函数作用

      • 阻塞并等待子进程退出
      • 回收子进程残留资源
      • 获取子进程结束状态(退出原因)

    • 返回值

      • 成功: 回收的子进程的 PID
      • 失败: -1 (没有子进程)

    • status 参数: 子进程的退出状态 – 这是供传出的参数,所以使用指针. 如果不想查status的状态,可以传NULL

      • WIFEXITED(status) : 为非0 –> 进程正常结束

        WEXITSTATUS(status) : 获取进程的退出状态

      • WIFSIGNALED(status) : 为非0 –> 进程异常终止

        WTERMSIG(status) : 取得进程终止的信号编号

  • waitpid()函数 -

    • 函数原型

      ​ pid_t waitpid(pid_t pid, int *status, int options);

    • 函数作用

      • 同 wait() 函数

    • 参数:

      • pid:

        ​ pid = -1 : 等待任一子进程.与wait()等效. 调用一次,只能回收一个

        ​ pid > 0 : 等待其进程ID与pid相等的子进程

        ​ pid = 0 : 等待进程组ID与目前进程相同的任何子进程. 也就是说任何和调用 waitpid()函数的进程在同一个进程组的进程

        ​ pid < -1 : 等待其组ID等于pid的绝对值的任一子进程. (适用于子进程在其他组的情况)

      • status : 子进程的退出状态. 用法同 wait()函数

      • options : 设置为 WNOHANG, 函数非阻塞; 设置为0, 函数阻塞.

    • 函数返回值

      • > 0 : 返回回收掉的子进程PID
      • = -1 : 无子进程
      • = 0 : 只有当参数3为 WNOHANG, 且子进程正在运行

  • 注意: 调用一次waitpid()函数或者wait()函数,只能回收一个子进程

    • 在回收多个子进程时, 一般可用 (返回值 == -1) 作为退出while()循环的条件

进程间通信概念

两个进程要想完成数据交换,必须通过内核. 一个进程将数据写到内核,然后另一个进程从内核读走数据. 内核提供的这种机制称为进程间通信( IPC: InterProcess Communication )

Linux下,经常用到的通信方式有: 文件、管道、信号、共享内存、消息队列、套接字、命名管道.

​ * 管道(使用最简单)

​ * 信号(开销最小)

​ * 共享映射区(无血缘关系)

​ * 本地套接字(最稳定)

管道

管道是一种最基本的IPC机制,也称匿名管道,应用于有血缘关系的进程之间,完成数据传递.调用 pipe()函数即可创建一个管道.

特质:

  • 管道的本质是一块内核缓冲区,内部使用环形队列实现
  • 管道有读写两端,读写两端是两个文件描述符,一个表示读端,一个表示写端.
  • 数据的流向是从管道的写端流入管道的读端(数据的流向是单向的)
  • 数据一旦被读走,在管道中就消失了
  • pipe只能用于有血缘关系的进程间通信
  • 管道的读端和写端默认都是阻塞的.
  • 管道的大小默认是4K.但是会根据实际情况做适当调整

创建管道 pipe()函数

  • 函数原型:

    ​ int pipe(int fd[2]);

  • 函数参数

    ​ 若函数调用成功,fd[0]存放管道的读端, fd[1]存放管道的写端

  • 返回值

    ​ 成功返回 0

    ​ 失败返回 -1,并设置 errno值

  • 用于父子进程间通信

    1. 父进程创建 pipe
    2. 父进程调用 fork()函数创建子进程
    3. 父进程关闭(不用的)一端
    4. 子进程关闭(不用的)一端
    5. 父进程和子进程分别执行read()和write()操作

管道的读写行为

  • 读操作
    • 有数据
      • read正常读,返回读出的字节数
    • 无数据
      • 写端全部关闭
        • read接触阻塞,返回0,相当于读文件读到了尾部
      • 写端没有关闭
        • read阻塞
  • 写操作
    • 读端全部关闭
      • 管道破裂,进程终止,内核给当前进程发 SIGPIPE信号
    • 读端没有全部关闭
      • 缓冲区写满了
        • write阻塞
      • 缓冲区没有满
        • 继续write

设置管道为非阻塞

设置读端或者写端为非阻塞,以下步骤

​ 第1步: int flags = fcntl(fd[0], F_GETFL, 0);

​ 第2步: flags |= O_NONBLOCK;

​ 第3部: fcntl(fd[0], F_SETFL, flags);

若是读端设置为非阻塞,则:

* 写端没有关闭,管道中没有数据可读, 则 read()返回-1;
* 写端没有关闭,管道中有数据可读, 则 read()返回实际读到的字节数
* 写端已经关闭,管道中有数据可读, 则 read()返回实际读到的字节数
* 写端已经关闭,管道中没有数据可读, 则 read()返回0

FIFO (命名管道)

FIFO是Linux基础文件类型的一种(文件类型为p, 可通过 ls -l查看到文件类型).

FIFO文件在磁盘上没有数据块,文件大小为0,仅仅用来标识内核中的一条通道.

进程可以打开这个文件进行 read()/write(), 实际上是在读写内核缓冲区,这样就实现了进程间通信.

创建FIFO管道

方式一: 使用命令 mkfifo

​ 命令格式: mkfifo 管道名

​ 例如: mkfifo myfifo

方式二: 使用函数

​ int mkfifo(const char *pathname, mode_t mode);

​ 参数说明和返回值,用命令查询: man 3 mkfifo

当创建了一个FIFO,就可以使用 open()函数打开它,常见的文件 I/O函数都可以用于FIFO. 如: close()、read()、write()、unlink()等

FIFO严格遵循先进先出(first in first out), 对FIFO的读总是从最开始处返回数据,而对它们的写则把数据添加到末尾.

注意: FIFO文件不支持诸如 lseek()等文件定位操作

FIFO完成两个进程间通信的思路:

  • 进程A:

    1. 创建一个 fifo文件: mkfifo命令或者使用 mkfifo()函数

      可以使用 access()函数来判断文件是否存在.若不存在,再行创建

      access()函数也可判断用户(或者进程)对该文件的权限(R_OK, W_OK, X_OK)

    2. open fifo文件, 获得一个文件描述符 fd

    3. 写 fifo文件: write(fd, “XXX”, …);

    4. 关闭 fifo文件: close(fd); 要等到进程B进行read()完毕后才能在进程A中予以关闭,否则会读不到?

  • 进程B:

    1. 打开 fifo文件, 获得文件描述符 fd (要迟于 进程A运行)
    2. 读 fifo文件: read(fd, buf, sizeof(buf));
    3. 关闭 fifo文件: close(fd);

内存映射区 mmap

  • mmap()函数

    • 作用:建立存储映射区

    • 函数原型:

      ​ void * mmap(void* adds, size_t length, int port, int flags, int fd, off_t offset);

    • 函数返回值:

      • 成功: 返回创建的映射区首地址
      • 失败: MAP_FAILED宏

    • 参数:

      • addr :指定映射的起始地址,通常设为NULL, 由系统指定

      • length :映射到内存的文件长度

        ​ 可用 lseek() 或者 stat()获取文件大小

      • prot :映射区的保护方式,最常用的有:

        • 读: PROT_READ
        • 写: PROT_WRITE
        • 读写: PROT_READ | PROT_WRITE

      • flags :映射区的特性,可以是

        • MAP_SHARED :写入映射区的数据会写回文件,且允许其他映射该文件的进程共享
        • MAP_PRIVATE :对映射区的写入操作会产生一个映射区的复制(copy-on-write),对此映射区所做的修改不会写回原文件

      • fd :由 open()返回的文件描述符,代表要映射的文件

      • offset :以文件开始处的偏移量. 必须是 4K的整数倍,通常为0,表示从文件开头开始映射

  • munmap()函数

    • 函数作用: 释放由 mmap()函数建立的存储映射区

    • 函数原型:

      ​ int munmap(void* addr, size_t length);

    • 返回值:

      • 成功: 返回0
      • 失败: 返回-1, 并设置了 errno只值

    • 函数参数:

      • addr :调用mmap函数成功返回的映射区首地址
      • length :映射区大小(mmap()函数的第二个参数)

  • 匿名映射: 只能用于有血缘关系的进程间通信

    ​ mmap(NULL, 4096, PORT_READ | PORT_WRITE, MAP_SHARED | MAP_ANONYMOUS, -1, 0);

信号概念

信号是信息的载体. 在Linux/Unix环境下,古老、经典的通信方式. 现在依然是主要的通信方式

特点:

​ 简单

​ 不能携带大量信息

​ 满足某个特定条件才会发生

信号的机制

进程A给进程B发信号,进程B收到信号之前执行自己的代码,收到信号后,不管执行到程序的什么位置,都要暂停运行,先去处理信号,处理完毕之后再继续执行原来的代码.这与硬件中断类似–异步模式. 但信号是软件层面上实现的中断,早期常被称为“软中断”

信号的优先级要高于普通操作.出现信号,要先处理

信号的状态

通过命令可以查看信号的种类: kill -l

信号有三种状态: 产生、未决 和 递达(已经被处理)

  • 信号的产生:
    • 按键产生,如: Ctrl+c、Ctrl+z、Ctrl+\
    • 系统调用产生,如: kill、raise、abort
    • 软件条件产生,如: 定时器 alarm
    • 硬件异常产生,如: 非法访问内存(段错误)、除0(浮点数例外)、内存对齐出错(总线错误)
    • 命令产生,如: kill命令
  • 未决: 产生和递达之间的状态.主要由于阻塞(屏蔽)导致了该状态
  • 递达: 递送并且到达进程
信号的处理动作
  • 执行默认动作
  • 忽略信号(丢弃不处理)
  • 捕捉信号(调用用户的自定义的处理函数)

信号的四要素:

  • 信号的名字(宏)

  • 信号的编号

  • 信号的描述(是如何产生的)

  • 信号的默认处理动作:

    • Term : 终止进程
    • Ign : 忽略信号(默认即时对该信种号忽略操作)
    • Core : 终止进程,生成Core文件(查验死亡原因,用于gdb调试)
    • Stop : 停止(暂停)进程
    • Cont : 继续运行进程

    特别需要注意: SIGKILL 和 SIGSTOP 这2个信号是不能被 捕获(caught)、阻塞(blocked)、忽略(ignored)的只能是执行默认动作

    默认动作:

    ​ SIGKILL: 杀死进程(Kill signal)

    ​ SIGSTOP: 暂停进程(Stop process)

信号的特质

信号的实现手段导致信号有很强的延时性,但对于用户来说,时间非常短,不易察觉.

Linux内核的进程控制块(PCB)是一个结构体(task_struct), 除了包含进程id、状态、工作目录、用户id、组id、文件描述符表, 好包含了信号相关的信息,主要是指阻塞信号集和未决信号集

  • 几个常用的信号: SIGINT, SIGQUIT, SIGKILL, SIGSEGV, SIGUSR1, SIGUSR2, SIGPIPE, SIGALRM, SIGTERM, SIGCHLD, SIGSTOP, SIGCONT

实际执行时间 = 系统时间 + 用户时间 + 损耗时间

损耗时间 = 实际执行时间 - ( 系统时间 + 用户时间 )

损耗原因: 调用printf()函数打印数字,遇到\n才会打印. 打印过程涉及到用户区切换到内核区,切换次数越多消耗的时间越长,效率越低; 而使用文件重定向,由于文件操作是带缓冲的,所以涉及到用户区切换到内核区的切换次数大大减少,从而使损耗降低,效率更高. 而使用全局变量,只在内存计数,仅使用信号触发函数打印,则将cpu效率发挥到极致.

setitimer()函数

  • 原型:

    ​ int setitimer(int which, const struct itimerval *new_value, struct itimerval *old_value);

  • 描述: 设置定时器(闹钟), 可替代alarm()函数,精度为微妙us, 可以实现周期定时.

  • 返回值:

    • 成功: 0;
    • 失败: -1, 并设置 errno值

  • 参数:

    • which : 指定定时方式

      • 自然定时: ITIMER_REAL 发送信号(14-SIGALARM), 计算自然时间(不管进程状态都计时)
      • 虚拟空间计时(用户空间): ITIMER_VIRTUAL 发送信号(26-SIGVTALRM), 只计算进程占用cpu的时间
      • 运行时计时(用户+内核): ITIMER_PROF 发送信号(26-SIGPROF), 计算占用cpu及执行系统调用的时间

    • new_value : struct itimerval, 负责设定 timeout时间

    • old_value : struct itimerval, 存放旧的 timeout值,一般指定为NULL

信号集相关

​ 当进程收到SIGINT信号后,首先被保留在 未决信号集中,并将标识位设为1.当这个信号被处理之前,先检查 阻塞信号集中对应的编号的位上的标识是否为1.

​ 为1则表示该信号被当前进程阻塞了.此时该信号暂时不被处理,对应的未决信号集中的位仍为1.

​ 若为0,则表示该信号没有被当前进程阻塞,则未决信号集中的这个信号需要被处理(通过忽略、执行默认动作、执行用户自定义函数).当信号被处理完毕后,解决信号集中的这个标识位从1变为0,表示该信号已经抵达了(已经被处理了).

未决信号集和阻塞信号集的数据类型都是 signet_t 在头文件<signal.h>中定义 ,按照信号的类型(编号)按位设置. 所以, 不能储存信号(即发生多次),只是按位判断. 如果发生了多次同样编号的信号,也仅是处理1次.

SIGCHLD信号

产生SIGCHLD信号的条件和作用

  • 条件: 当发生以下情况时, 内核会向父进程发送一个SIGCHLD信号
    • 子进程结束的时候
    • 子进程收到SIGSTOP信号时
    • 当子进程停止时,收到SIGCONT信号
  • 作用
    • 当子进程退出后,内核会向父进程发送一个SIGCHLD信号. 父进程收到这个信号后可以对子进程进行回收.
    • 使用SIGCHLD信号完成对子进程的回收,可以避免父进程阻塞等待而不能执行其他操作.只有当父进程收到SIGCHLD信号之后才去调用捕捉函数完成对子进程的回收,未收到SIGCHLD信号之前可以处理其他操作

练习: 父进程创建三个子进程,然后让父进程捕获SIGCHLD信号完成对子进程的回收

注意点:

  • 有可能还未完成信号处理函数的注册,三个子进程就都退出了
    • 解决办法: 可以在fork()前先将SIGCHLD信号阻塞.当完成信号处理函数的注册后再解除阻塞
  • 当SIGCHLD信号函数处理期间,SIGCHLD信号若再次产生是被阻塞的,而且若产生多次,则该信号只会被处理一次,这样可能产生僵尸进程
    • 解决办法:可以在信号处理函数中使用while(1)循环回收,这样就有可能出现捕获一次SIGCHLD信号但是可以回收多个子进程的情况,从而可以避免产生僵尸进程.当waitpid()函数返回-1时就表示已经回收了所有的子进程了,此时可以break了;若waitpid()函数返回0时,表明当时没有僵尸进程(所有剩余子进程都在运行态)了,可以break并返回到父进程继续运行.
    • 父进程在退出时,是否需要检查是否还有子进程活着,并予以处理(等待或者杀死或发出退出信号)?

进程组和会话

进程组

  • 使用命令: kill -SIGKILL -进程组ID(负值) 可以将整个进程组内的进程全部杀死

  • 一个会话是一个或多个进程组的集合
  • 创建会话的进程不能是进程组组长

  • 进程组和会话

    • 会话: 多个组组成一个会话
  • 创建会话的进程不能是组长进程
  • 一般创建会话的是父进程.要先fork()子进程,然后父进程退出,让子进程调用setsid()函数创建一个会话.这样,这个子进程即是会长,也是组长
  • 只要创建了会话,这个进程就脱离了控制终端的印象

线程

轻量级的进程(LWP: light weight process), 在Linux环境下线程的本质仍是进程.

- 主线程和子线程谁先执行? 不一定,谁先抢到cpu资源,谁先执行

    • thread : 传出参数,保存系统为我们分配好的线程ID.

      ​ 在Linux中,可以理解为: typedef unsigned long int pthread_t

    • attr : 通常传NULL,表示使用线程默认属性.若想使用具体属性也可修改该参数

    • start_routine : 函数指针,指向线程主函数(线程体).该函数结束运行,则线程结束

    • arg : 线程主函数执行期间所使用的参数

    • 由于pthread_create()函数的错误码不保存在errno中,因此不能直接用perror()打印出错误信息.可以先用strerror()把错误码转换成错误信息再打印

pthread_exit()函数

  • 描述: 将单个线程退出. 主线程调用此函数也不会使整个进程退出,也不影响其他线程的执行. 在线程中禁止调用exit()函数

pthread_detach()函数

线程分离状态: 指定该状态,线程主动与主控线程断开关系.线程结束后,其退出状态也不由其他线程获取,而是直接自己释放自己.网络、多线程服务器常用.

进程若有该机制,将不会产生僵尸进程.僵尸进程的产生主要由于进程死后,大部分资源被释放,但仍有一点残留资源存在系统中,导致内核认为该进程仍存在.

也可使用pthread_create()函数参数2(线程属性attr)来设置线程分离. pthread_detach()函数是在创建线程后调用的.

  • 描述: 实现线程分离

  • 原型: int pthread_detach(pthread_t thread);

  • 返回值:

    • 成功: 0;
    • 失败: 错误号

  • 注意: 一般情况下,线程终止后,其状态一直保留到其它进程调用pthread_join()函数获取状态为止.但是**线程也可以被置为detach状态,这样线程一旦终止就立刻回收它暗涌的所有资源,而不保留终止状态**.不能对一个已经处于detach状态的线程调用pthread_join()函数,这样的调用将返回EINVAL错误.也就是说,如果对一个线程调用了pthread_detach()函数后,就不能再调用pthread_join()函数了.

  • 练习: 编写程序,在创建线程之后设置线程为分离状态

互斥锁

线程A和线程B共同访问共享资源.当线程A想访问共享资源的时候,要先获得锁.

如果锁已经被占用,则加锁不成功,需要等待对方释放锁;

如果锁没有被占用,则获得锁成功(加锁),然后才开始操作共享资源.操作完之后,必须解锁.同理,进程B也同样操作.

也就是说,同时不能有两个线程同时访问共享资源,属于互斥操作.

原子操作: 该操作要么不执行,要么就完成.

死锁

死锁是因为代码不规范,操作不当造成的

  • 第一种情况
    • 自己锁自己
    • 加锁后没有解锁
    • 异常情况退出,退出前没有解锁. 注意,这很重要,在异常退出前,一定要先解锁
  • 第二种情况
    • 2把锁,分别被2个线程分别锁定,而且都没有释放自己的锁,又都想去锁对方的锁造成的
    • 要避免使用嵌套的锁

解决死锁的办法

  • 所有线程按照一定的顺序访问共享资源,要避免使用嵌套的锁
  • 在访问其他锁的时候,需要想解开自己的锁
  • 调用 pthread_mutex_trylock()函数,如果锁不成功会立刻返回

读写锁 (是一把锁)

也叫共享-独占锁. 写独占,读共享

使用场景: 使用读写锁非常适用于对数据结构读的次数远远大于写的情况.

特性:

  • 已经以写模式加锁时 : 解锁前,所有对该锁加锁的线程都会被阻塞
  • 已经以读模式加锁时 : 如果线程以读模式再加锁时会成功, 但要求以写模式加锁时,会被阻塞
  • 已经以读模式加锁时 : 既有试图以写模式加锁要求的线程,也有试图以读模式加锁要求的线程, 那么读写锁会阻塞随后的读模式加锁请求,优先满足写模式加锁要求. 即 读锁和写锁并行阻塞时,写锁优先级高.

    • 参数:
      • rwlock : 读写锁
      • attr : 读写锁属性,传NULL为默认属性

  • 销毁读写锁: int pthread_rwlock_destroy(pthread_rwlock_t * rwlock);

  • 加读锁: int pthread_rwlock_rdlock(pthread_rwlock_t * rwlock);

  • 尝试加读锁: int pthread_rwlock_tryrdlock(pthread_rwlock_t * rwlock);

  • 加写锁: int pthread_rwlock_wrlock(pthread_rwlock_t * rwlock);

  • 尝试加写锁: int pthread_rwlock_trywrlock(pthread_rwlock_t * rwlock);

  • 解锁: int pthread_rwlock_unlock(pthread_rwlock_t * rwlock);
